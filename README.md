# Text Game Agent

Text Game Agent 是一个本地文字游戏 agent，用来把故事资料、人物状态和玩家选择持续转化成可游玩的小说正文。

它不是酒馆式 chatbox，也不是一套更长的提示词预设。它把互动小说拆成多次、有分工的模型调用：先判断剧情该怎么推进，再写正文，再给出下一轮剧情方向，同时由程序维护长期状态和历史压缩。

## 相比酒馆的优势

酒馆的核心模式是单次长提示词：角色卡、世界书、历史记录、规则、风格、禁忌和玩家输入都会被塞进同一次模型调用里，让模型一次性完成理解、规划、写作、记忆和自我修正。

Text Game Agent 的核心模式是多阶段文字游戏 agent。程序负责拆任务、保存状态、压缩历史、调度模型；模型每次只处理一个相对明确的任务。

### 1. 多次调用替代单一长提示词

单一长提示词会让模型同时承担太多工作：理解设定、回忆历史、规划剧情、写正文、避免重复、更新状态。提示词越长，注意力越分散，小模型尤其容易顾此失彼。

Text Game Agent 把一轮生成拆成多个任务：导演负责判断本轮剧情方向和过渡，叙述者负责正文，选项策略器负责下一轮方向，总结器负责后台记忆更新。每次调用的目标更窄，模型更容易做好当前任务。

这意味着它不完全依赖单个强模型硬扛全部工作。通过程序拆分任务和多次调用，较弱模型也能接近更强模型在单次长上下文里才有的整体效果。

### 2. 内置审查抑制文字重复

LLM 文游很容易自我重复：同样的动作、眼神、沉默、心理解释、暧昧气氛和原地拉扯，会在后续轮次里不断强化。

Text Game Agent 在生成正文前内置审查器，专门检查文字细节重复、剧情设计重复和剧情速度拖沓。它不会等玩家抱怨后再修，而是在本轮正文生成前就把整改要求交给叙述者。

这能减少常见的文游问题：同一场景写太久、对话没有推进、动作反复描摹、人物关系一直停在试探状态。

### 3. 长期变量由程序维护

酒馆主要靠上下文里的文本记忆维持一致性。上下文变长后，人物状态、性格、关系、位置、世界观规则很容易漂移，旧设定也容易被新文字覆盖。

Text Game Agent 把长期变量放在程序内维护，例如世界观、人物状态、人物性格、关键记忆、物理约束和关键信息。模型写正文，程序负责把稳定事实保存下来，再在下一轮重新注入。

这能降低失忆和漂移：人物不会只因为几轮没出现就消失，关系变化不会只靠模型模糊记忆，位置、姿势、物理限制也不会完全依赖自然语言上下文碰运气。

### 4. 历史正文分层压缩

长篇文游不能无限把历史正文塞进上下文。正文越长，模型越容易失焦：重要事件被稀释，近期场景被远期文本干扰，旧细节也未必能准确召回。

Text Game Agent 会把历史正文压缩成分层摘要：近期正文保留为热记忆，较早轮次压缩成事实总结，更长历史再分段压缩。需要旧细节时，再由召回模块选择相关旧轮次。

这样做的目的不是假装摘要无损，而是让上下文保持可控：当前轮次优先关注当前场景，同时保留长期剧情、关系变化和重要事实的导航能力。

### 5. 候选项从高层规划剧情方向

传统文游选项常常是即时动作或台词，例如“继续问她”“走过去”“沉默等待”。这种选项会让剧情一直贴着当前场景滑行，很容易拖慢节奏。

Text Game Agent 的候选项不是角色台词或动作，而是未来剧情方向。选项策略器从更高层级生成 5 个方向，类型包括 `推进`、`转折`、`跳过`。玩家选择的是“剧情往哪里走”，不是“角色下一句说什么”。

这种设计让候选项成为真正的高层导演：它可以推动关系变化、引入信息压力、快速跳到第二天或下一个剧情节点，避免剧情在同一场景里无休止地寒暄、试探和等待反应。

## 基本流程

```text
玩家输入或选择剧情方向
  -> Director 判断本轮推进和场景过渡
  -> Narrator 输出小说正文
  -> OptionStrategist 生成下一轮剧情方向
  -> 玩家继续选择或自由输入

后台：
  -> Summary 更新人物状态、记忆和历史总结
  -> RecallWorker 预取可能需要的旧正文
```

## 运行

```bash
npm install
npm run web
```

默认地址：

```text
http://127.0.0.1:4173
```

## 当前目标

Text Game Agent 的目标不是替代人类小说设计，也不是模拟传统游戏的即时操控感。它更适合处理传统游戏难以细腻表达的部分：人物关系变化、对话压力、长期情绪后果、信息误判和剧情方向选择。

它希望把 LLM 文游从“长提示词聊天”推进到“有程序骨架的互动小说系统”：正文由模型生成，长期状态由程序维护，剧情方向由高层候选项控制。
