# Text Game Agent

Text Game Agent 是一个本地文字游戏 agent，用来把故事资料、人物状态、玩家输入持续转化成可游玩的小说正文。

它不是普通聊天框，也不是单纯的提示词预设。它把文字游戏拆成多个职责清晰的 agent 模块：导演、叙述者、总结器、反馈器。每轮游戏不是把一整坨长 prompt 扔给模型，而是让不同模块分别处理“剧情怎么动”“正文怎么写”“记忆怎么留”“问题怎么修正”。

## 和酒馆的区别

酒馆本质上是文游 chatbox。

它的核心能力来自提示词工程：把角色卡、世界书、预设、历史记录和各种规则拼进一次模型调用里，让模型一次性完成理解、规划、写作、记忆和自我修正。

Text Game Agent 的目标是文游 agent。

它把 harness 思想融入文字游戏：程序不只是拼 prompt，而是维护一个多阶段生成闭环。模型每次只做一个相对明确的任务，程序负责保存状态、压缩记忆、调度模块、把结果回灌到下一轮。

## 核心优势

### 1. 多模块调用，而不是单一长 prompt

酒馆通常是一轮调用完成所有事情。prompt 越堆越长，模型注意力会被规则、风格、历史、状态、禁忌和输出要求分散。

Text Game Agent 把一次复杂生成拆成多个简短 prompt：

```text
玩家输入
  -> Director
  -> PlanFeedback
  -> Narrator
  -> 玩家可见正文 + 候选项

后台：
  -> PostprocessSummary
```

这样每个模型调用的目标更窄，注意力更集中。

### 2. 双流水线，降低体感延迟

文字游戏最怕每轮等待太久。Text Game Agent 把主生成链路和后处理链路拆开：

```text
主链路：Director -> PlanFeedback -> Narrator -> 立即返回正文和候选项
后台链路：PostprocessSummary -> 更新长期总结和人物状态
```

玩家看到正文后就可以继续输入。长期总结和人物状态可以滞后一轮更新。

这比把总结、检查、反馈都塞进同一次调用里更适合实际游玩：主链路更短，体感速度更快，系统仍然能在后台维护长期一致性。

### 3. 导演和叙述者分离

Director 不写正文，只负责本轮剧情推动和写法调度。

当前导演计划核心字段：

```json
{
  "plotDrive": "剧情推动力；如何推动",
  "mainPresentation": "主要呈现方式",
  "supportingPresentation": ["辅助呈现方式"],
  "narrativeStyle": "本轮动态叙事风格",
  "physicalConstraints": ["本轮纯物理约束"]
}
```

Narrator 只负责把导演计划写成玩家可见正文。

这个分工的好处是：Director 可以动态判断这一轮应该用对话推进、动作推进、心理描写、环境烘托、一笔带过等方式推动局面；Narrator 则保留足够自由度，负责具体文字、细节和节奏落点。

相比直接用 beat 限死剧情，这种结构更适合释放 LLM 的叙事能力。

### 4. 候选项降低输入负担

Narrator 每轮除了正文，还会生成 3 个玩家候选输入。

这些候选项不是传统选项树，也不是强制分支，只是给玩家偷懒用的下一步行动建议。玩家可以直接点选；如果愿意，也可以完全自由地输入任何行动、对话或想法。

这样可以降低文字游戏的输入负担：用户不需要每一轮都重新想“我该说什么、做什么”，但自由输入始终是完整保留的。

### 5. 负反馈模块抑制重复和拖沓

模型天然有自我重复倾向。已经出现过的动作、句式、桥段、情绪反应，会在后续生成里被继续强化。

Text Game Agent 在 Narrator 写正文前运行 PlanFeedback，检查：

- `文字细节重复`
- `剧情设计重复`
- `剧情速度拖沓`

它不写正文，也不读取最近导演计划；只根据最近正文和本轮 Director 计划生成整改要求。旧 Director 计划不算叙事事实，避免过去的导演意图继续束缚 Narrator。

负反馈直接注入本轮 Narrator，让叙事者当轮整改。

如果剧情连续停留在寒暄、等待、移动、解释、重复反应或原地对峙上，PlanFeedback 会要求本轮加快剧情速度，必要时使用 `一笔带过` 或直接引入新压力、新信息、新场景阶段。

这是单一 prompt 很难稳定做到的，因为单一 prompt 没有独立的后处理视角。

### 6. 长期一致性由程序维护

文字游戏最容易出问题的是长期一致性：

- 人物失忆
- 性格漂移
- 姿势错位
- 空间关系混乱
- 世界观前后矛盾
- 已发生事件被重复发生
- 物理约束被忽略

Text Game Agent 把这些信息拆开维护：

- 世界观
- 人物状态
- 最近正文
- 历史总结
- 物理约束
- 玩家负反馈
- 后处理反馈

最近正文作为热记忆直接进入下一轮；历史总结作为冷记忆定期压缩；人物状态由 PostprocessSummary 根据最终正文更新；物理约束由 Director 逐轮继承。

## 当前架构

```text
Initializer
  读取故事资料
  生成世界观、开场正文、初始人物状态、初始选项

Director
  读取玩家输入、最近正文、人物状态、世界观、历史总结、上轮反馈、物理约束
  输出本轮剧情推动力、落成局面、呈现方式、动态叙事风格、物理约束

Narrator
  读取导演计划和 PlanFeedback
  写玩家可见正文
  输出正文和 3 个玩家选项

PostprocessSummary
  读取最终正文
  更新本轮事实总结和人物状态补丁

PlanFeedback
  在 Narrator 前检查重复、拖沓和剧情设计路径依赖
  给本轮叙事者提供反重复和剧情加速反馈
```

## 设计原则

- 程序是 backbone，不是死板状态机。
- Director 不写正文，不生成细纲 beat。
- Narrator 不维护状态，只写正文。
- PlanFeedback 不写正文，只服务本轮整改。
- PostprocessSummary 不改正文，只服务长期记忆和人物状态。
- 长期变量只保存稳定事实和必要锚点。
- 动态风格由 Director 每轮生成，不由人类预设死。
- 具体情节和细节留给 Narrator 自由发挥。

## 运行

```bash
npm install
npm run web
```

默认地址：

```text
http://127.0.0.1:4173
```

## 近期待办

1. 提供一个案例故事。
   案例故事要能让用户直接上手，包含故事资料、人物状态、开场配置和推荐玩法入口，避免用户启动后不知道玩什么。

2. 研究不同故事的导入与初始化。
   不能只在同一个故事上反复调试。需要用多种题材、结构和资料格式的故事测试导入流程，验证 Initializer 能否稳定生成世界观、人物状态、开场配置和初始候选项，让不同故事都能进入可玩状态。

3. 打包成 Electron App。
   目标是不再要求用户手动在网页里启动服务，而是下载/打开 App 后即可配置 DeepSeek Key、导入故事并开始游玩。

4. 补教程和宣传视频。
   教程说明如何配置、导入案例故事、初始化、输入行动、使用候选项、保存/读取存档。宣传视频说明 Text Game Agent 是什么、和酒馆的区别、为什么导演/叙述者/总结/反馈分层更适合文字游戏。

5. 设计高级导演模块和剧情库。
   初始化阶段预先生成剧情库，每个剧情单元包含准入条件、剧情大纲正文、结束条件。后处理判断当前是否需要新的剧情大纲；需要时调用高级导演查阅剧情库，选出或生成下一个剧情大纲，再注入低级导演，让低级导演负责本轮具体推进。

目标：做到新用户能轻易启动这个项目，并在几分钟内开始玩起来。
