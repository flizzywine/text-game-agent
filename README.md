# Text Game Agent

Text Game Agent 是一个本地文字游戏 agent，用来把故事资料、人物状态、玩家输入持续转化成可游玩的小说正文。

它不是普通聊天框，也不是单纯的提示词预设。它把文字游戏拆成多个职责清晰的 agent 模块：导演、叙述者、选项策略器、总结器。每轮游戏不是把一整坨长 prompt 扔给模型，而是让不同模块分别处理“剧情怎么承接”“正文怎么写”“下一轮往哪里走”“记忆怎么留”。

## 和酒馆的区别

酒馆本质上是文游 chatbox。

它的核心能力来自提示词工程：把角色卡、世界书、预设、历史记录和各种规则拼进一次模型调用里，让模型一次性完成理解、规划、写作、记忆和自我修正。

Text Game Agent 的目标是文游 agent。

它把 harness 思想融入文字游戏：程序不只是拼 prompt，而是维护一个多阶段生成闭环。模型每次只做一个相对明确的任务，程序负责保存状态、压缩记忆、调度模块、把结果回灌到下一轮。

## 核心优势

### 1. 多模块调用，而不是单一长 prompt

酒馆通常是一轮调用完成所有事情。prompt 越堆越长，模型注意力会被规则、风格、历史、状态、禁忌和输出要求分散。

Text Game Agent 把一次复杂生成拆成多个简短 prompt：

```text
玩家输入
  -> Director
  -> Narrator
  -> OptionStrategist
  -> 玩家可见正文 + 候选项

后台：
  -> Summary

旁路：
  -> RecallWorker
```

这样每个模型调用的目标更窄，注意力更集中。

### 2. 双流水线，降低体感延迟

文字游戏最怕每轮等待太久。Text Game Agent 把主生成链路和后处理链路拆开：

```text
主链路：Director(内置审查) -> Narrator -> OptionStrategist -> 立即返回正文和候选项
后台链路：Summary -> 更新长期总结和人物状态
旁路链路：RecallWorker -> 预取旧正文轮次缓存
```

玩家看到正文后就可以继续输入。长期总结和人物状态可以滞后一轮更新。召回也不阻塞主链路，只提前准备下一轮可能用到的最多两轮旧正文。

这比把总结、检查、反馈都塞进同一次调用里更适合实际游玩：主链路更短，体感速度更快，系统仍然能在后台维护长期一致性。

### 3. 候选项定方向，导演做执行和过渡

OptionStrategist 生成的候选项已经框定下一轮宏观剧情方向。Director 不写正文，也不再重新设计未来剧情，只负责把用户实际选中的那一条方向落成本轮剧情推动、场景审查和写法调度；未选中的候选方向不进入 Director。

当前场景的主要信息、关系变化或阻力已经兑现时，Director 应主动快进到入睡、第二天、下一个地点或下一个剧情节点，不继续在同一场景上加码。

当前导演计划核心字段：

```json
{
  "sceneTransition": "从上一轮到本轮的场景切换/过渡依据",
  "sceneOutcome": "本轮结束时必须已经成立的新局面",
  "narrativeStyle": "本轮叙事风格",
  "文字细节重复": "",
  "剧情设计重复": "",
  "剧情速度拖沓": "",
  "叙事整改要求": ""
}
```

Narrator 只负责把导演计划写成玩家可见正文，不能输出导演字段、标签或说明。

这个分工的好处是：候选项决定战略方向，Director 负责把方向变成可执行的过渡和落点，Narrator 保留足够自由度，负责具体文字、对白、细节和节奏落点。

相比直接用 beat 限死剧情，这种结构更适合释放 LLM 的叙事能力。

### 4. 候选项选择剧情方向

Narrator 只生成正文。正文之后由 OptionStrategist 读取长历史，生成 5 个候选剧情方向。

这些候选项不是角色台词、动作、即时回应或完整下一幕情节，也不是传统选项树。它们是中层剧情方向，可以点明矛盾类型、关系变化、信息压力或节奏去向，类型只允许 `推进`、`转折`、`跳过`。玩家点击后提交的是“剧情方向选择”，只有被选中的这一条由下一轮 Director 转译成本轮计划，不直接进入正文。

这样可以降低文字游戏的输入负担：用户不需要每一轮都重新想“我该说什么、做什么”，而是选择希望剧情往哪个方向走；自由输入仍然保留。

### 5. Director 内置审查抑制重复和拖沓

模型天然有自我重复倾向。已经出现过的动作、句式、桥段、情绪反应，会在后续生成里被继续强化。

Text Game Agent 不再单独运行 Feedback。Director 在生成计划时内置检查：

- `文字细节重复`
- `剧情设计重复`
- `剧情速度拖沓`

它不写正文，只根据最近正文、用户输入和候选方向判断本轮是否该推进、转折或跳过，避免过去的气氛和动作继续束缚 Narrator。

内置审查字段会随 Director 计划一起注入本轮 Narrator，让叙事者当轮整改。

如果剧情连续停留在寒暄、等待、移动、解释、重复反应或原地对峙上，Director 会要求本轮加快剧情速度，必要时使用 `一笔带过` 或直接引入新压力、新信息、新场景阶段。

这是单一 prompt 很难稳定做到的，因为单一 prompt 没有独立的后处理视角。

### 6. 初始化配置可热更新，长期一致性由程序维护

文字游戏最容易出问题的是长期一致性：

- 人物失忆
- 性格漂移
- 姿势错位
- 空间关系混乱
- 世界观前后矛盾
- 已发生事件被重复发生
- 物理约束被忽略

Text Game Agent 把这些信息拆开维护：

- 世界观
- 人物状态
- 最近正文
- 历史总结
- 物理约束
- 玩家负反馈
- Director 内置审查

最近正文作为热记忆直接进入下一轮；历史总结作为冷记忆；人物状态、关键信息和物理约束由 Summary 根据最终正文更新。

初始化配置里的 `worldview / directorStyle / narratorStyle / openingSummary` 属于稳定配置。你可以直接编辑 `story/<storyId>/program-config.json`，保存后下一轮自动生效。人物状态、状态字段和长期变量属于运行态，不会被初始化配置覆盖。

## 当前架构

```text
Initializer
  读取故事资料
  生成世界观、开场正文、初始人物状态、初始选项

Director
  读取玩家输入、最近正文、人物状态、世界观、历史总结、上轮反馈、物理约束
  输出 sceneTransition、sceneOutcome、narrativeStyle 和内置审查

Narrator
  读取导演计划和 Director 内置审查
  以全知第三人称写玩家可见正文
  只输出正文

OptionStrategist
  读取正文、导演计划、内置审查和长历史
  输出 5 个中层剧情方向

Summary
  读取最终正文
  更新本轮事实总结和人物状态补丁
```

## 设计原则

- 程序是 backbone，不是死板状态机。
- Director 不写正文，不生成细纲 beat；它只做方向执行和场景审查。
- Director 必须给出 `sceneTransition`，让每轮正文之间有自然过渡。
- Narrator 不维护状态，只写正文；正文使用全知第三人称，允许切换焦点人物。
- 不恢复独立 Feedback；重复、拖沓和整改要求属于 Director 内置审查。
- Summary 不改正文，只服务长期记忆和 `longTermState`。
- 长期变量 `longTermState` 保存人物状态、关键信息和物理约束，只保留稳定事实和必要锚点。
- 世界观、导演风格、叙事风格可在 `program-config.json` 中手动修改，下一轮自动生效。
- 具体情节和细节留给 Narrator 自由发挥。

## 运行

```bash
npm install
npm run web
```

默认地址：

```text
http://127.0.0.1:4173
```

   案例故事要能让用户直接上手，包含故事资料、人物状态、开场配置和推荐玩法入口，避免用户启动后不知道玩什么。

2. 研究不同故事的导入与初始化。
   不能只在同一个故事上反复调试。需要用多种题材、结构和资料格式的故事测试导入流程，验证 Initializer 能否稳定生成世界观、人物状态、开场配置和初始候选项，让不同故事都能进入可玩状态。

3. 打包成 Electron App。
   目标是不再要求用户手动在网页里启动服务，而是下载/打开 App 后即可配置 DeepSeek Key、导入故事并开始游玩。

4. 补教程和宣传视频。
   教程说明如何配置、导入案例故事、初始化、输入行动、使用候选项、保存/读取存档。宣传视频说明 Text Game Agent 是什么、和酒馆的区别、为什么导演/叙述者/总结/反馈分层更适合文字游戏。

5. 设计高级导演模块和剧情库。
   初始化阶段预先生成剧情库，每个剧情单元包含准入条件、剧情大纲正文、结束条件。后处理判断当前是否需要新的剧情大纲；需要时调用高级导演查阅剧情库，选出或生成下一个剧情大纲，再注入低级导演，让低级导演负责本轮具体推进。

6. 增加“剧情引导”项目。
   用户可以写入自己希望剧情靠近的方向，例如想看关系修复、揭露阴谋、离开当前场景、推进某条人物线。该内容输入给 Director，作为长期但可修改的软方向，让导演每轮朝剧情引导靠拢，避免剧情持续滑向用户不想看的方向。当前不做。

7. 补齐“继续未完成”的阶段断点。
   当前继续未完成能复用 Director 计划；Narrator、OptionStrategist、Summary、RecallWorker 还没有完整断点缓存。后续应做成按阶段恢复：已经完成的阶段可见保存，刷新或失败后从下一个未完成阶段继续。这个缓存只用于同一轮故障恢复，不做跨轮命中复用。

8. 研究整本小说的分级记忆树。
   L0 保留原始正文切块，L1 每隔约 1500 字压缩一次，L2 每 10 份 L1 再压缩一次，继续向上堆成树。摘要层只负责压缩和导航，不假装无损；每个节点必须保留 children/source_l0 指针，让系统能从高层下钻到 L0，也能从原文回溯到上层。后续可加检索 LLM：当 Director/Narrator 缺失细节时，先查高层定位候选区间，再下钻到对应 L0 原文，把原文证据注入下一轮。当前不做，先作为长篇小说级记忆方案。

9. 删除右侧“上轮反重复提醒”展示。
   该信息属于内部反馈状态，保留给 Director 使用即可；前端右侧不需要常驻展示，避免挤占状态栏空间。

目标：做到新用户能轻易启动这个项目，并在几分钟内开始玩起来。
