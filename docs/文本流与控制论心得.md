# 文本流与控制论心得

文字游戏不是世界模拟器，而是一条连续生成的文本流。

人物状态、剧情总结、当前目标都应该服务于“让下一段文字接得上上一段文字”。它们是记忆锚点，不是数据库模型。做状态维护，不是为了把程序控制插入文字游戏世界，而是给 LLM 一个记忆锚点，使它在输出长文本流时维持前后连贯，不至于涣散。字段里偶尔写入不完全符合定义的内容，并不必然是错误；只要它能让模型记住人物关系、情绪余波、身体处境、场景惯性，它就是有效的。

过度管理会让 agent 停滞。飞机流程、道具位置、姿势变化、好感度数值一旦被显式建模，模型就会把维护这些状态当成任务本身。结果是剧情反复滑行、起飞、攀升，卡在物理状态切换上，而不是向人物关系和事件变化推进。

Director 也不应该承担建模世界的职责。导演的作用不是维护一套世界状态机，而是给文本流一副骨架：本轮要有怎样的压力、误判、反应、转折和未解窗口。骨架让流体不散掉，但不能把流体冻成机械流程。

更好的方式是控制论，而不是管理论。

让模型自由输出文本流，再用反馈控制方向。重复了，就反馈重复；节奏慢了，就反馈推进不足；违反物理约束了，就反馈物理错误；人物失真了，就反馈 OOC；长期目标失焦了，就反馈导演没有靠近剧情目标。

状态的价值不在于精确建模，而在于让文本流带一点粘性。它让前后文不至于断裂，让人物不突然重置，让已经发生的事留下痕迹。状态越像规则引擎，越容易拖慢剧情；状态越像压缩记忆，越能帮助文本自然流动。

长期变量也不是程序变量，不该用 if/else 的方式理解，而是记忆种子和承前启后的骨架。它回灌给 prompt，不是为了让 LLM 像执行代码一样精确遵守规则，而是给文本流一个持续影响方向的锚点。它应该承前：保留已经发生的关键事实、人物关系、状态变化；也应该启后：给下一轮一个可继续生长的方向；还要防散：避免每轮像重新开局一样漂移、重复、断裂。它连接短期正文和长期走势，重要的是前后连贯，不是被逻辑条件锁死。

不能用确定性程序思维管理长期变量。写了规则，不等于模型会一丝不苟执行；声明禁用词，不等于正文不会出现；设置剧情目标，不等于每轮都会机械靠近。LLM 会响应约束，但不会服从到状态机级别。长期变量要短，像种子，不像合同；给方向，不堆命令。具体长出什么，要交给模型在当轮上下文里自然生长。

硬约束如果反复失败，不要继续堆 prompt。禁用词、重复、物理错误这类问题，更适合交给后处理检测、下一轮质量提醒，或未来的修订轮。工程要做的是播种、修剪和反馈，不是用规则表去接管文本生命。

关键不在固定规则，而在动态约束。

物理约束、文风约束、反重复约束，都不是能提前完整写死在 prompt 里的东西。它们来自正文已经发生的变化：人物现在站在哪里、身体处在什么姿势、刚刚用过什么桥段、上一轮是什么语气、当前场景已经消耗了多少耐心。这些约束只有在文本流动之后才会产生，必须由 Director、PlanFeedback 和状态总结逐轮提取、压缩、回灌。

固定 prompt 只能给底线和方向，不能预先列出所有当轮有效约束。真正有效的控制来自“读上一段文字，生成这一轮约束”。文游 agent 的核心能力，不是提前设计一套万能规则，而是让文字变化本身不断生成新的物理边界、风格边界和反重复边界。

程序也不应该被理解成 LLM 的 harness 或 controller，而应该被理解成 backbone。它不是为了控制 LLM，而是收集 LLM 已经输出的文本、状态和方向，必要时压缩，再反复注入给下一轮。它的目的不是让模型服从程序，而是让模型自己的输出前后连续、一致、有承接。

backbone 的工作是承重和连接：把上一轮正文、人物状态、剧情方向、反重复提醒、物理约束这些骨架材料保存下来，再送回生成流里。LLM 仍然负责生长、联想和表达；程序只负责让这条文本流不散架、不失忆、不每轮重启。

文游的核心不是“把世界管住”，而是“让文字流动，并用反馈修正流向”。

## 导演不写剧情清单

通过导演动态生成叙事风格、节奏、呈现方式、叙事策略、镜头落点和语言手法，比让导演输出 beats 更有效。

beats 看起来像剧情控制，实际会把叙事者绑成执行器。导演一旦给出 `beat1 -> beat2 -> beat3`，叙事者就会倾向于逐条完成、逐条扩写、逐条补细节，结果是细节过密、节奏变慢、文字像任务清单，不像自然流动的小说。

更好的分工是：导演不直接生成剧情链，也不替叙事者写正文；导演只给本轮推进点、剧情驱动力、主要呈现方式、辅助呈现方式、叙事策略、节奏、镜头和语言风格。这些是动态约束，不是剧情清单。

剧情细节、段落组织、具体反应、对白密度、停顿位置和文本生命力，应该交给叙事者自由发挥。导演负责“这轮为什么动、朝哪里动、用什么方式动”；叙事者负责“它具体怎么活起来”。

这种结构释放了 LLM 的临场能力：长期变量保证连贯，短期目标保证方向，动态风格保证质感，具体剧情和文字流交给叙事者生成。文游玩的就是变化，规则如果把每一段剧情都预先绑死，就退回了传统文字小说的写法。

## 候选项是速度调节器

候选项不是简单的“给用户省输入”，也不只是延续当前场景的自然尾巴。候选项本身就是剧情速度的重要调节器。

如果候选项只追求连续和连贯，它会把玩家继续引回原场景、原对峙、原试探，叙事者也会顺着慢节奏继续写。剧情拖沓时，候选项必须主动承担加速功能：一条延续当前场景但必须产生变化，一条让玩家操控人物主动收束当前场景，一条引入第三方或外部变化，把场景推进到新阶段。

候选项应该帮助用户偷懒，但不能替剧情踩刹车。它应该把玩家输入导向“变化”，而不是导向“继续等待”。

## 反馈前移

把反馈放在正文之后，只能做到下一轮纠偏。本轮已经写慢了，后处理再发现拖沓，也只能把提醒塞进未来上下文。

更有效的结构是把反馈前移到导演计划之后：

```text
Director -> PlanFeedback -> Narrator
```

PlanFeedback 先审查导演计划和最近正文，判断是否继续原地对峙、重复桥段、细节过密或剧情拖沓。问题还没进入正文前，就直接把整改要求注入 Narrator。

这样拖延不会先写成正文，再等待下一轮修正；它会在叙事层生成前被拦住。Narrator 接到的是“导演计划 + 本轮整改要求”，能当轮一笔带过低价值过程，直接让新压力、新信息、新关系变化或新场景阶段成立。
