# 文本流与控制论心得

文字游戏不是世界模拟器，而是一条连续生成的文本流。

人物状态、剧情总结、当前目标都应该服务于“让下一段文字接得上上一段文字”。它们是记忆锚点，不是数据库模型。做状态维护，不是为了把程序控制插入文字游戏世界，而是给 LLM 一个记忆锚点，使它在输出长文本流时维持前后连贯，不至于涣散。字段里偶尔写入不完全符合定义的内容，并不必然是错误；只要它能让模型记住人物关系、情绪余波、身体处境、场景惯性，它就是有效的。

过度管理会让 agent 停滞。飞机流程、道具位置、姿势变化、好感度数值一旦被显式建模，模型就会把维护这些状态当成任务本身。结果是剧情反复滑行、起飞、攀升，卡在物理状态切换上，而不是向人物关系和事件变化推进。

Director 也不应该承担建模世界的职责。导演的作用不是维护一套世界状态机，而是给文本流一副骨架：本轮要有怎样的压力、误判、反应、转折和未解窗口。骨架让流体不散掉，但不能把流体冻成机械流程。

更好的方式是控制论，而不是管理论。

让模型自由输出文本流，再用反馈控制方向。重复了，就反馈重复；节奏慢了，就反馈推进不足；违反物理约束了，就反馈物理错误；人物失真了，就反馈 OOC；长期目标失焦了，就反馈导演没有靠近剧情目标。

状态的价值不在于精确建模，而在于让文本流带一点粘性。它让前后文不至于断裂，让人物不突然重置，让已经发生的事留下痕迹。状态越像规则引擎，越容易拖慢剧情；状态越像压缩记忆，越能帮助文本自然流动。

文游的核心不是“把世界管住”，而是“让文字流动，并用反馈修正流向”。
