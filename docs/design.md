# Text Game Agent 设计

## 目标

把酒馆预设里的自然语言模块工程化：

```text
系统硬规则
导演
叙事者
编辑
后处理
用户配置项
动态加载项
```

除此之外不再保留其他 prompt 分类。

## 设计原则

先构造可运行闭环，再让真实运行暴露优化方向。

当前阶段不提前优化想象中的复杂问题。不要在系统还没有稳定跑起来时，预设过多边界、例外、冲突仲裁和最优策略。

优先级：

```text
先能 run
-> 玩多轮
-> 观察真实坏点
-> 根据事实修正架构
-> 再做性能、缓存、复杂边界和精细策略
```

暂时不优先处理：

- 用户强行突破长期剧情约束怎么办。
- 长期规划和本轮计划冲突怎么办。
- 情节库如何最优检索。
- 长期剧情如何精确判断完成。
- 伏笔如何跨几十轮精准回收。
- 缓存如何最大化命中。

这些属于真实运行后再处理的问题。提前优化可能解决的是不存在的问题，也可能把系统锁进错误方向。

当前真正要保证的是：

- 长期剧情能生成。
- Director 能参考并更新长期剧情。
- Narrator 能写出可读正文。
- Director 能决定伏笔 plant/delay/payoff/drop，程序能合并伏笔队列。
- Postprocess 能更新人物状态、环境状态、事实总结、写作负反馈和选项。
- 前端能展示并允许玩家继续输入。

目标不是一次设计完美架构，而是尽快获得事实反馈。事实反馈比脑内推演更可靠。

## Prompt 目录

```text
prompts/
  system-hard-rules/
    hard-rules.md
    must-read-top.md
    must-read-bottom.md
  director/
    prompt.md
  narrator/
    prompt.md
  postprocess/
    prompt.md
  user-config-templates/
    *.md
  dynamic-loading/
    README.md
```

各 prompt 文件统一格式：

```text
用途：这个 prompt 负责什么
什么时候用：在哪一层、哪一步调用
怎么用：程序如何读取和注入
---
实际发送给模型的 prompt 正文
```

程序只读取 `---` 之后的正文。

## 固定流水线

每轮默认生成三次调用：

```text
Director -> Narrator -> Postprocess
```

| 层 | 文件 | 职责 |
|---|---|---|
| Director | `prompts/director/prompt.md` | 剧情规划、模块化剧情、描写发展链、伏笔、节奏、玩家窗口 |
| Narrator | `prompts/narrator/prompt.md` | 根据导演计划写正文；内化 OOC、全知、时间线、世界观、重复、禁用项、推进过快、空间物理可行性等写作纪律 |
| Postprocess | `prompts/postprocess/prompt.md` | 不修改正文；生成本轮事实总结、写作负反馈、人物状态、环境状态、玩家候选输入 |

这三层都是固定模块，不是 skill。Narrator 正文完成后前端先显示可读正文，Postprocess 继续后台更新状态、候选项和 `qualityFeedback`。玩家发起下一轮输入前，上一轮 Postprocess 必须完成。

`sceneState` 是唯一的环境状态字段。它只保存一句会持续约束行动合法性的场景事实，例如载具阶段、天气、场所开放状态、公共规则或正在发生的外部事件。它不是状态机，不拆字段；Director 和 Narrator 只把它当作场景约束，Postprocess 每轮沿用或更新。

## 系统硬规则

`prompts/system-hard-rules/` 保存全局硬规则：

- `must-read-top.md`：包裹每层 prompt 的上边界。
- `must-read-bottom.md`：包裹每层 prompt 的下边界。
- `hard-rules.md`：每层注入的不可覆盖规则。

`必读上` 和 `必读下` 不混入普通硬规则，而是包裹每层完整 prompt。

## 用户配置项

用户配置项是玩家手动启用的偏好，不由 Director 动态请求。

```text
prompts/user-config-templates/*.md
```

一个配置项就是一个 Markdown 文件。文件头保存元数据，分隔线之后保存实际注入内容：

```md
---
id: "细腻叙事"
name: "细腻叙事"
description: "更贴近感受，动作和神态替代空泛形容。"
group: "叙事风格"
exclusiveGroup: "narrator-style"
layer: "narrator"
enabled: false
---
这里写实际注入内容。
```

后端每次处理 `/api/modules` 时只扫描 `prompts/user-config-templates/`。旧 `prompts/user-config/` 保留在文件系统里，但运行时当它不存在；新增或删除配置项，直接增删新目录里的 `.md` 文件，网页刷新后生效。

当前只开放两类用户配置：

| 配置层 | 作用 |
|---|---|
| 导演风格 | 控制剧情如何新增、矛盾如何处理、人物关系如何变化、谁更主动、长期剧情倾向 |
| 叙事风格 | 控制正文语言质感、视角距离、人称、句子节奏、描写比例、禁用表达 |

启用后按 `layer` 注入：

```text
layer: director -> Director
layer: narrator -> Narrator
Postprocess 不接收用户配置
```

后处理不做风格配置。Postprocess 只使用固定通用 prompt，保持事实总结、人物状态、写作负反馈和玩家选项稳定。

导演风格和叙事风格都是单选。即使多个文件同时写了 `enabled: true`，后端也只保留同层排序后的第一个生效，避免风格互相拉扯。

## 动态加载项

动态加载项是真正可由 Director 请求加载的知识库内容。

来源：

- 酒馆人物卡 JSON / PNG
- 故事书 JSON / Markdown / txt
- 世界书 / World Info JSON
- 手动新增故事条目
- Postprocess 生成的长期剧情总结
- 最近正文窗口

当前 MVP 运行方式：

```text
前端导入内容
-> 存入当前故事槽 / localStorage
-> 生成时把启用条目发给 /api/generate
-> 后端渲染为【已加载动态注入模块】
```

每个故事槽独立保存人物状态、故事资料、全局上下文、最近正文、用户模块开关和候选项。保存存档会导出当前故事 JSON；加载存档会把 JSON 作为新故事加入并切换过去。

Director 现在会看到轻量 registry，也可以输出 `loadModuleIds`。程序暂时还没有执行二阶段按需加载；当前仍是“启用条目整体注入”。

下一步动态加载目标：

```text
Director 先看 registry
-> 输出 loadModuleIds
-> 程序只加载被请求的知识库条目
-> Narrator 使用精简后的动态内容
```

## 长篇小说导入：剧情原型库

长篇小说导入不是把原作改成互动版，而是把原作编译成可迁移的叙事素材。

初始化时先从小说开头生成传统故事配置：

```text
小说开头
-> 世界观
-> 人物卡
-> 初始人物关系
-> 当前局面
-> 开场白
-> 初始玩家选项
```

小说后续内容不作为固定剧本，不要求复刻原作路线。后续内容应转成“剧情原型库”：

```text
小说后续
-> 原作事件实例
-> 剧情原型
-> 角色槽位
-> 前置条件
-> 戏剧功能
-> 变形规则
-> 禁用条件
```

原作不是铁路，而是素材矿。系统要最大程度利用现成小说中的精彩创意，同时保留玩家巨大自由度。

### 剧情原型

情节库不应只保存固定事件，例如：

```text
A 在雨夜误会 B 背叛自己。
```

应抽象成可迁移原型：

```text
一方看到不完整证据，误判另一方背叛、隐瞒或动机不纯。
```

同一原型可以迁移到不同角色和地点：

```text
原作：A 误会 B
运行时：A 误会 C，或 C 误会 A
```

剧情原型示例：

```json
{
  "id": "plot_proto_012",
  "prototype": "不完整证据导致误会",
  "sourceExample": "原作中 A 在雨夜看到 B 与第三方密谈，误以为 B 背叛。",
  "dramaticFunction": "制造信任裂缝",
  "roles": {
    "misreader": "误会者",
    "target": "被误会者",
    "trigger": "不完整证据",
    "thirdParty": "可选第三方"
  },
  "requiredConditions": [
    "误会者在意目标",
    "双方已有信任或依赖",
    "目标存在未说明的信息或行为"
  ],
  "mutationRules": [
    "角色槽位可替换为当前关系紧张的人物",
    "地点可替换",
    "证据可替换为物件、称呼、目击、聊天记录、能力暴露",
    "误会强度可从轻微怀疑到关系破裂"
  ],
  "cannotUseWhen": [
    "双方完全无关系",
    "关键信息已经公开",
    "会强行覆盖玩家刚做出的选择"
  ]
}
```

运行时优先使用剧情原型，原作事件实例只作为风味和参考。

### 剧情动力模型

导演层的核心剧情要素分为四类：

| 要素 | 生效时间 | 职责 |
|---|---|---|
| 长期剧情 | 多轮之后 | 给出远期方向、关系弧线、主线压力和核心矛盾 |
| 短期剧情 | 本轮 / 近 1-3 轮 | 承接当前局面，让角色和事件缓慢靠近长期剧情 |
| 伏笔 | 未来回收 | 延迟生效的扰动，负责把宏观方向埋进微观细节 |
| 主动扰动 | 本轮立刻生效 | 即时生效的扰动，负责打破平衡、制造选择和压力 |

长期剧情和短期剧情是宏观 / 中观设计。伏笔和主动扰动是微观动力。宏观必须用微观实现：长期剧情不能只写“信任裂缝加深”，必须落到物件、动作、台词、环境变化、第三方动静、信息差或 NPC 反应上。

伏笔和主动扰动同属“剧情动力”，区别在生效时间：

```text
伏笔 = 现在埋下或延迟，未来回收
主动扰动 = 现在出现，立刻改变局面
```

Director 每轮工作顺序：

```text
1. 查看当前长期剧情目标
2. 判断长期剧情是否已完成、失效或被玩家改写
3. 若为空、完成、失效或偏离太远，直接生成、修订或替换当前长期剧情
4. 若未完成，设计本轮短期剧情，让局面向长期目标靠近一点
5. 选择微观动力：
   - 伏笔：plant / delay / payoff
   - 主动扰动：立刻制造变化
6. 设计结尾玩家窗口
```

长期剧情不保存完成条件。完成与否交给 LLM 在每轮自行判断，避免把灵活剧情固化成任务清单。

长期剧情示例：

```json
{
  "goal": "A 对 B 的信任开始破裂",
  "pressure": "亲近表象与身份隐瞒冲突",
  "direction": [
    "让亲近动作里逐渐出现试探",
    "用第三方信息制造不完整证据"
  ]
}
```

当长期剧情完成后，Director 直接输出更新：

```json
{
  "mode": "replace",
  "longArc": {
    "goal": "承接信任裂缝，转入双方互相试探的关系压力",
    "pressure": "想靠近与不敢坦白并存",
    "direction": ["让解释变成新的试探", "引入第三方不完整证据"],
    "prototypes": ["试探", "误会", "隐瞒"]
  }
}
```

Director 返回新的长期剧情更新：

```json
{
  "longArc": {
    "goal": "A 暗中试探 B，而 B 试图维持表面亲近",
    "pressure": "亲近表象与暗中试探并存",
    "direction": ["把关心写成试探", "让隐藏信息以小物件形式回响"],
    "prototypes": ["误会", "试探", "隐瞒"]
  }
}
```

剧情推进链条：

```text
长期剧情 A
-> 多轮短期剧情缓慢靠近
-> 伏笔 / 主动扰动推动局面变化
-> Director 判断旧长期剧情已达成、失效或偏离太远
-> 长期剧情 B
```

这个模型替代单纯“大纲规划”。不再按 1-20 轮、21-40 轮规划；当前长期剧情只保存方向和压力，完成判断和修订都交给 Director。

### 合并导演层

当前阶段不单独保留 Story Curator。长期剧情规划和本轮节拍规划合并进 Director，一次调用内完成。

| Director 内部职责 | 说明 |
|---|---|
| 长期剧情管理 | 当前长期剧情为空时生成；已达成、失效或偏离时修订/替换 |
| 本轮计划 | 管三段节拍、扰动、伏笔、结尾窗口、场景约束 |

Director 分两种素材模式：

```text
有情节库：
Director = 原作情节筛选器 + 原型变形器 + 长期剧情规划器 + 本轮节拍规划器

无情节库：
Director = 原创长期剧情规划器 + 戏剧压力设计器 + 原型生成器 + 本轮节拍规划器
```

当前阶段先实现“无情节库模式”。Director 读取世界观、人物关系、历史总结、人物状态和当前局面，在第一轮正式玩家输入后临场生成当前长期剧情。

无情节库时，Director 的长期剧情更新仍然输出统一结构：

```json
{
  "longArc": {
    "goal": "未来 10-20 轮要缓慢抵达的剧情变化",
    "pressure": "推动剧情发展的核心矛盾",
    "direction": [
      "可逐步靠近的方向1",
      "可逐步靠近的方向2"
    ],
    "prototypes": [
      "误会",
      "试探"
    ]
  },
  "plotForces": [
    {
      "type": "foreshadow",
      "function": "延迟生效，未来回收"
    },
    {
      "type": "disturbance",
      "function": "立即生效，打破当前平衡"
    }
  ],
  "usablePrototypes": [
    {
      "prototype": "误会 / 试探 / 隐瞒 / 第三方介入 / 共同危机 / 旧事重现",
      "function": "制造关系变化、暴露信息或推进长期目标"
    }
  ]
}
```

Director 不关心长期剧情材料来自原作、情节库还是临场原创。接口保持一致，后续加入情节库时只替换素材来源，不改 Narrator 和 Postprocess。

Director 不直接加载整本小说。Director 发现当前轮缺少长程素材、原作事件、伏笔背景、后续压力或新的长期目标时，可以在 `load` 中请求剧情原型库条目。

请求格式先保持简单：

```json
{
  "load": ["plot-prototype.trust-misread"],
  "brief": "当前互动缺少外部压力；需要一个不直接揭露秘密但能制造怀疑的事件。"
}
```

剧情原型库返回少量可用材料：

```json
{
  "materials": [
    {
      "prototypeId": "plot_proto_012",
      "prototype": "不完整证据导致误会",
      "mapping": {
        "misreader": "C",
        "target": "A",
        "trigger": "A 藏起的旧信件",
        "thirdParty": "B"
      },
      "adaptedBeat": "C 无意看见 A 藏起旧信，误以为 A 和 B 仍有秘密联系。",
      "constraints": [
        "不能坐实背叛，只制造怀疑",
        "给 A 留解释窗口"
      ]
    }
  ],
  "discarded": [
    {
      "prototypeId": "plot_proto_004",
      "reason": "当前地点和人物关系已偏离原作前置条件"
    }
  ],
  "pressure": "信任与隐瞒的矛盾开始升温"
}
```

Director 拿到的不是原文，而是这包可用剧情材料。Director 再决定本轮是否使用、用多少、如何落地、放在哪个节拍、是否回收伏笔以及给玩家留下什么窗口。

### MVP 原则

先做能 run，不提前做性能优化。

第一版流程：

```text
1. 小说导入时指定开头范围和后续范围
2. 开头转成普通故事配置
3. 后续转成剧情原型库
4. Director 读取当前长期剧情；若为空、已达成、失效或偏离太远，直接输出 `longRangeUpdate`
5. Director 生成本轮计划
6. Narrator 写正文
```

先保证链路可运行，再逐步做缓存、预热、复杂检索和自动性能优化。

## 后续优化

### 下一轮预热缓存

已实施基础版。

思路：在当前轮 Postprocess 完成后，程序可以基于最新全局上下文、人物状态、当前长期剧情和 3 个玩家候选项，提前生成候选 Director / Narrator 缓存。下一轮如果玩家点击候选项，或自由输入能匹配某个缓存计划，只需命中缓存后继续 Postprocess，减少等待时间。

约束：

- 缓存只用于“下一轮候选方向”，不能锁死玩家自由输入。
- 玩家自由输入明显偏离缓存计划时，必须丢弃缓存，重新调用 Director。
- Postprocess 未完成前不得预生成，因为人物状态、历史总结、写作负反馈还没落定。
- 缓存计划必须带版本号，绑定 story id、turn index、globalContext hash、statusPanel hash、longRangeOutline hash，避免旧计划误用。
- 缓存命中只优化速度，不改变固定流水线的职责边界。
- 玩家发起新输入时，前端取消正在等待的预热请求。
- 后端生成当前输入时，取消所有不匹配的后台预热任务；匹配中的预热只等待 `PREWARM_WAIT_MS`，超时后取消并转前台生成，避免页面卡死。

## 可待能力内化

| 可待能力 | 工程位置 |
|---|---|
| 要求强化 | Director 规划，Narrator 执行，Postprocess 校验状态 |
| 全面小说创作指导 | Director / Narrator / Postprocess 分层内化 |
| 模块化剧情 | Director `sceneBeats` / `endingBeat` |
| 描写发展链 | Director `focus` |
| 推进规范 | Director 控节奏，Narrator 执行慢推进 |
| 伏笔、长期剧情 | Director 规划伏笔动作和长期剧情更新，程序合并伏笔队列 |
| 实时总结 | Postprocess 固定输出 |
| 逻辑加强 | Director/Narrator 正向约束，Postprocess 只输出写作负反馈 |
| 自我辩证 | Postprocess 输出可废弃的写作负反馈，下一轮注入 Director / Narrator |
| 禁用库 | 硬规则和 Narrator 内化，用户可额外扩展 |
| 文生图 | 保留为 Postprocess 扩展点 |

不要求模型输出完整思维链，只保留结构化检查字段和最终可用数据。
