# 记忆分层

当前记忆分为四层：`L0`、`L0.5`、`L1`、`L2`。层级越低，越接近原文和当前正文；层级越高，越粗、越稳定、越适合长期方向。

## L0：正文层

L0 是原始正文和玩家输入，尽量不解释、不压缩。

当前变量：

| 变量 | 内容 | 来源 | 用途 |
| --- | --- | --- | --- |
| `recentTurns` | 最近正文，包含用户输入和正文 | 前端从当前消息流截取 | 给 Director 提供最近 1 轮上下文连接；给 Feedback 和 Narrator 看近处连续性、重复和刚发生的场景阶段 |
| `story.txt` | 每轮玩家输入 + 最终正文 | `appendRawTurnLog()` 追加 | 作为可召回的完整原文 |
| `raw-turns.jsonl` | 每轮结构化原始记录 | `appendRawTurnLog()` 追加 | 调试和结构化排查 |
| `recallEvidence` | RecallWorker 已完成缓存里的问题-答案对 | RecallWorker 提问并锁定旧轮次、程序加载对应 `story.txt` 完整轮次、RecallQA 根据证据回答后写入 `recall-cache.json` | 注入 Narrator，用来补远处细节，但不注入完整旧正文 |
| `playerInput` | 本轮玩家输入，统一视为剧情方向选择 | 前端请求 | Director、Narrator、Summary 的当前触发点 |
| `finalText` | 本轮最终正文 | Narrator 输出 | Summary 的主要总结依据 |

原则：

- L0 最具体，也最占上下文。
- 最近 L0 只给需要具体文字的模块。
- 远处 L0 不常驻，只通过召回读少量轮次。
- 总结模块只看本轮 `playerInput` 和 `finalText`，不读远处 L0。

## L0.5：状态层

L0.5 是随着正文变化而变化的状态。它介于 L0 和 L1 之间：比原文更短，比事实总结更结构化，主要用于保持当前可行动条件和实体连续性。

当前变量：

| 变量 | 内容 | 来源 | 用途 |
| --- | --- | --- | --- |
| `statusSchema` | 人物状态字段 | Initializer 初始化，Summary 可补字段 | 约束人物状态结构 |
| `statusRoster` | 追踪人物名列表 | Initializer 初始化，Summary 可补人物 | 防止人物重复、别名漂移 |
| `longTermState.characterStatus` | 人物当前位置、姿势、情绪、外显状态等 | Initializer 初始化，Summary 逐轮更新 | 给 Director / Narrator / Summary 维持人物连续性 |
| `longTermState.keyItems` | 关键道具的持有人、位置 | Initializer 初始化，Summary 逐轮更新 | 防止手机、钥匙、信件等关键道具在多人/多地点之间漂移 |
| `longTermState.physicalConstraints` | 纯物理限制 | Summary 根据最终正文逐轮更新 | 给 Director / Narrator 避免物理连续性错误 |
| `planFeedback` | 本轮整改计划，包含重复、拖沓和叙事整改要求 | Director 内置审查生成 | 当轮注入 Narrator，要求正文立即避让重复、加速或调整写法 |
| `feedbackMemory` | 上轮反重复、拖沓等整改提醒 | Director 内置审查后前端短期保存 | 给下一轮 Director 作为临时控制反馈，避免刚出现的问题立刻复发 |
| `longTermState.keyInfo` | 关键信息状态 | Summary 逐轮维护 | 保存持续重要但不适合放入人物状态、物品状态或逐轮 L1 的信息 |
| `controlledCharacterName` | 当前操控人物 | 开局选择或前端状态 | 限定“我”的指向，辅助状态更新 |

原则：

- L0.5 不是完整世界状态机，只保存会影响下一轮正文的当前状态。
- 状态随正文变化而变化，必须由本轮正文驱动，不从远处世界观或旧总结强行改写。
- LLM prompt 统一读取 `longTermState`，不再把人物状态、关键道具、关键信息、物理约束拆成多个输入块。
- 旧字段 `statusState`、`itemState`、`physicalConstraints` 保留为存档兼容和前端同步字段；运行时会合并进 `longTermState`。
- `longTermState.characterStatus` 管人物，`longTermState.keyItems` 管关键道具；不要扩展成完整物品系统。
- `longTermState.physicalConstraints` 只写纯物理限制，不写人物意图、关系目标或剧情建议。
- `planFeedback` 和 `feedbackMemory` 都是 L0.5。它们是控制生成流的自动整改状态，不是世界设定，也不是长期事实总结；`planFeedback` 只在当轮注入 Narrator，`feedbackMemory` 短期自动保存给下一轮 Director。
- 状态裁剪是未来优化。后续可以只加载在场人物、当前交互相关人物、当前场景可触达或剧情相关的关键道具，减少上下文；当前暂不实现。
- `longTermState.keyInfo`（关键信息）属于 L0.5，不是 L1；用于保存跨轮持续重要但不适合写进人物状态、物品状态或逐轮事实总结的信息，例如当前隐含目标、关键误会、关系暗线、场景内未解决的重要事实。它必须随正文变化增删改，不能无限增长。

## L1：事实总结层

L1 是每轮一两条事实总结，格式保持 `第X轮：总结内容`。它比 L0 粗，但保留轮次定位，可用于长期连续性和召回索引。

当前变量：

| 变量 | 内容 | 来源 | 用途 |
| --- | --- | --- | --- |
| `turn-summaries.txt` | 每轮事实总结 | Summary 追加或替换同轮总结 | 全量 L1，不能删除 |
| `turn-summaries-l2.txt` | 每 10 轮压成一条的区间总结 | SummaryL2 在整十轮生成 | L2 历史压缩索引，不能删除 |
| `longHistoricalSummary` | 排除最近 5 轮后的长历史；第 25 轮开始用 L2 替换 1-10 轮 L1，第 35 轮替换 11-20 轮，之后每隔 10 轮推进 | 后端从 `turn-summaries.txt` 和 `turn-summaries-l2.txt` 程序层渲染 | 给 Director/RecallWorker 看更少条数的长期历史 |
| `shortHistoricalSummary` | 排除最近 5 轮后的最近 10 条 L1 | 后端从 `turn-summaries.txt` 选择 | 给 Narrator 看短期历史，避免前言不搭后语 |
| `turnSummary` | 本轮事实总结 | Summary 输出 | 写入 L1 |

原则：

- L1 文件不删除、不覆盖；L2 文件也不删除。压缩只发生在 `longHistoricalSummary` 的程序渲染层。
- RecallWorker 和 Director 读取替换后的长历史，用于判断长期阶段和旧因果。
- Narrator 只读取短 L1，不默认看远。
- 独立 Feedback 已删除；重复、拖沓和整改要求由 Director 内置审查负责。
- Summary 不读取 L1；它只生产 L1。

## L2：设定与风格层

L2 是最稳定、最粗的背景和风格。它不随每轮正文频繁变化，主要提供世界边界、故事基调和写法方向。

当前变量：

| 变量 | 内容 | 来源 | 用途 |
| --- | --- | --- | --- |
| `worldview` / `storyContext` | 世界观、背景、开局前已成立规则 | 故事库 `program-config`，开局复制到当前故事 | 给 Director 判断宏观边界和长期方向 |
| `directorStyle` | 导演风格 | 故事库配置 / 当前故事配置 | 给 Director 和 Feedback 约束推进方式 |
| `narratorStyle` | 叙事风格 | 故事库配置 / 当前故事配置 | 给 Narrator 和 Feedback 约束正文语气 |
| `openingText` | 开场白原文 | 故事初始化 | 作为第 0 轮写入 L0 |
| `openingSummary` | 开场白压缩总结 | 初始化配置或开局时生成 | 作为第 0 轮写入 L1 |

原则：

- L2 稳定、粗，不应替代当前正文事实。
- Director 可以看 L2，因为它负责宏观方向。
- 世界观很短但信息密度高，可以作为短 L2 导向给 Narrator、Feedback、Summary。
- Narrator 读取世界观只为理解故事边界、基调和知识限制，不把它当成远处正文。
- Feedback 读取世界观只为判断合理性和风格边界，不读取长历史。
- Summary 读取世界观只为理解语境，不得把设定写成本轮事实。

## 模块使用矩阵

| 模块 | L0 | L0.5 | L1 | L2 |
| --- | --- | --- | --- | --- |
| HighLevelDirector | 不读 | 可读少量宏观状态，不处理细节 | 读全量 L1 或更高层粗摘要 | 读世界观、导演风格、剧情库 |
| RecallWorker | 读最近正文；先锁定远处旧轮次，再加载对应 `story.txt` 完整正文回答 | 读人物状态、关键道具、物理约束、关键信息 | 读替换后的长历史 | 读短世界观 |
| Director | 只读最近 1 轮 L0；不输出召回字段 | 读人物状态、关键道具、物理约束、反馈状态 | 读全量长 L1 | 读世界观、导演风格 |
| Narrator | 读最近正文；读已完成的 RecallWorker 问答缓存 | 读人物状态、关键道具、物理约束、本轮 `planFeedback` | 读短 L1 | 读短世界观、叙事风格 |
| Summary | 只读本轮玩家输入和最终正文 | 读当前状态并输出 patch | 不读旧 L1，只输出本轮 `turnSummary` | 读短世界观作为语境 |
| Initializer | 生成开场 L0 | 生成初始状态 | 生成第 0 轮总结 | 生成世界观和风格 |

## 高级导演定位

高级导演是比当前 Director 更粗、更远的一层，不是替代低级 Director。

未来流水线可以变成：

```text
HighLevelDirector -> Director -> Feedback -> Narrator
```

高级导演只负责宏观剧情方向：当前处于哪个长期阶段、是否需要切换剧情大段、哪条关系线或冲突线应该被推进、是否需要引入新的剧情压力。它读取更远的 L1/L2，必要时读取剧情库或更高层摘要，但不读取 L0 原文，不写本轮正文，不输出细节 beat。

低级 Director 仍然负责本轮动态计划：把高级导演给出的宏观方向落成本轮 `plotDrive`、`sceneOutcome`、呈现方式和叙事风格。召回由旁路 RecallWorker 负责，物理约束由 Summary 根据最终正文更新。

这样分层后，高级导演看得更远但更粗；低级 Director 看得中等，负责本轮推进；Narrator 看得最近，负责具体文字。

## RecallWorker

召回已经从 Director 和前台主链路拆出，不再使用 `recallTurns`。

当前旁路：

```text
正文输出 / Summary 完成 -> RecallWorker -> recall-cache.json
```

`RecallWorker` 只做一件事：根据替换后的长历史、最近正文和必要 L0.5，提出下一轮附近可能会用到的旧知识问题，并锁定需要回看的旧正文轮次；程序加载这些轮次的完整 `story.txt` 正文；RecallQA 再根据完整轮次输出问题-答案对，写入 `recall-cache.json`。

它不阻塞 Director、Narrator 或 Summary。下一轮 Narrator 只读取已经完成的缓存；缓存不存在时当作无召回。当前不写剧情建议，不做一致性检查，不把完整旧正文注入后续模块。

每次 RecallWorker 完成或报错都写入 `recall-worker-events.jsonl`，前端全局流水线通过轮询展示这些 JSON。刷新页面时，前端先读取当前轮已完成事件；如果当前轮没有事件，会调用后端重新触发 RecallWorker，再继续轮询。这样浏览器刷新或服务器重启不会让召回结果永久丢失。

## 当前结论

导演看得最远，但看得最粗，并只读取最近 1 轮 L0 连接上下文；反馈者看得中等，粗细都看，但范围要近；叙事者默认只看近，远处细节通过召回少量 L0；总结模块看得最近，只把本轮事实写入 L1 和 L0.5。

不要把同一份上下文平均塞给所有模块。记忆分层的目的不是压缩越狠越好，而是让每个模块只拿完成职责所需的最低层级。
