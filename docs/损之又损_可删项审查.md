# text-game-agent 可删项审查

按"为道日损"的方法过一遍代码库，找出可以删除的累赘。

当前规模：
- `scripts/web-server.ts` 3070 行 / 122 函数
- `web/app.js` 3675 行 / 208 函数
- `src/jsonObjectParser.ts` 117 行
- `web/index.html` 275 行 / `web/styles.css` 1283 行
- 合计 ~8420 行

最近 5 个 commit 都是 add/refine/backfill，没有任何一次是纯删除。这是为学路径，不是为道路径。

---

## 一级可损：明显冗余

### 1. 服务端和前端重复实现同一组状态面板函数

`ensureSpatialStatusPanelSchema`、`ensureSpatialStatusPanel`、`ensureMultiPersonStatusPanelSchema`、`ensurePlayerStatusPanelForPrompt`、`stripSpatialStatusSection`、`trackedCharacterNames`——这 6 个函数在 `scripts/web-server.ts` 和 `web/app.js` 各写一遍，逻辑几乎一致。

风险：两边一旦不同步就是隐形 bug。

**损法**：选一边删。状态面板归一化只在生成 prompt 时需要——前端不需要，让前端拿服务端归一化后的结果就好。直接删 `web/app.js` 里的 6 份重复实现，约 ~150 行净删。

### 2. 4 个 writeXxxDebugFile 函数本质同一件事

`writeLlmDebugFile`、`writeEvaluationMaterialFile`、`writeEvaluationDebugFile`、`writeCodexEvaluationRequestFile`——都是"加时间戳、拼路径、写 JSON、写 latest.json"。

**损法**：合成一个 `writeDebugArtifact(kind, payload)`。约 100 行 → 30 行。

### 3. 三个 Request 接口大量重叠字段

`GenerateRequest`(22 字段)、`PostprocessRequest`(24 字段)、`EvaluationRequest`(extends + 5 字段)——大半字段一样，每加一个字段要在多处同步。

**损法**：抽 `PipelineContext` 基础接口，三个 Request 各自只列差异字段。

## 二级可损：过度设计的抽象

### 4. Fireworks priority 命名舞蹈

`fireworksDeepSeekV4ProModel` / `fireworksDeepSeekV4ProPriorityModel` / `fireworksRequestModel()` / `fireworksServiceTier()` 一堆函数只为表达"是不是 priority 版本"。但 `modelIds` 里只注册了 priority 版本，普通版本是死代码路径。

**损法**：删掉 priority/非 priority 的区分，写死 priority。或反过来删非 priority 的所有引用。

### 5. Provider 抽象只有 2 个 provider，且 deepseek 路径不一定在用

`providerForModel`、`providerLabel`、`providerBaseUrl`、`providerApiKey`、`providerHasApiKey`、`isFireworksModel` 撑一个二选一。Default model 已经写死 fireworks priority。

**损法**：确认 deepseek 那条路径是否还在使用。若实际只跑 fireworks，整层 provider 抽象删掉，~80 行 + 一组判断分支。典型的"为以后可能用"的过度抽象。

### 6. 迷你 YAML parser

`readPrompt` / `readPromptFile` / `parseMarkdownConfig` / `unquoteYamlValue`——为了从 `prompts/*.md` 读 frontmatter 写了一个迷你 YAML parser。

**损法**：看 frontmatter 实际复杂度。如果只是 key: value 单行，10 行能搞定，不需要这套。

---

## 三级可损：架构层面

### 7. `block()` 被调用 86 次拼 prompt

每个 stage 的 prompt 组装都是 `[block('A', a), block('B', b), ...].join('\n')`。把 prompt 模板拆成了散落各处的字符串拼接，可读性差，改一处要改多处。

**损法**：让 prompts 目录的 `.md` 文件本身就是模板（mustache 风格 `{{characters}}`），代码里只做"读模板 + 注入变量"。所有 `block()` 调用消失。这是把"形而下的拼接"还给"形而上的模板"。

### 8. fallback 链太长

`normalizeProgramConfig` 每个字段都是 `String(raw.x || fallback.x || '')`。代码不信任输入，也不信任默认值，处处兜底。但兜底本身是复杂度——它把"什么是合法配置"这个契约打散了。

**损法**：定义清晰的 schema（zod 或手写），输入要么合法要么报错。fallback 集中在一处，不要散落几十处。

## 四级可损：默默存在的累赘

### 9. 顶层全局集合

`internalizedModuleIds`、`metadataKeys` 这种顶层 Set 经常是早期某个特殊处理的残留。审一下还在不在被读取、是否仍然必要。

### 10. 122 个函数在一个文件里

不是必须拆，但 3070 行单文件 + 122 函数意味着没有清晰边界。给每个函数标上"如果删了，谁会哭"——估计 1/3 死了没人哭。

## 不要动的部分

- **pipeline 主流程**（initializer / director / narrator / postprocess 的分阶段）。这是项目的"道"，不能损。
- **`jsonObjectParser`**。核心工具，独立性好，无依赖牵连。

## 三刀 ROI 排序

按性价比排，最值得先动手：

1. **删前端那 6 个重复函数**——30 分钟，去掉一类隐形 bug 风险
2. **合并 4 个 writeXxxDebugFile**——30 分钟，~70 行净删
3. **prompts 改模板文件，删 `block()` 拼接**——半天，从根本上改善可改性

## 元判断

这个项目已经比大部分代码库干净。但代码层面还没进入损的节奏——最近 5 个 commit 全是加法。

**下一次迭代不是加新功能，而是花一个下午只删不加，看能减多少行**。

这是检验自己有没有真正在为道的最直接方法。损卦初爻"酌损之"，二爻"弗损益之"——损要趁早，第二爻就该踩刹车。这个项目刚到 8000 行，正是动手的时候。
